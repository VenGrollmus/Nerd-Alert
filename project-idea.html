Project Description

Overview

Topic

In this project, the team will plan and design the tool we proposed in A2, a piece of software that can be used to empower staff in an organisation by deploying software with conversational interactions, which is essentially a chatterbot. The chatterbot we are designing is based on open-source libraries, which will allow us to customise the User Experience and how the chatterbot operates.

A chatbot in an organisation can be a valuable tool to support staff within HR, Finance, and Staff Resources on services offered, process information, legislation, and policy details offered at the organisation.

We aim to design the chatbot to allow the user to input questions or queries conversationally. The chatbot will process the input using the nltk module and find an answer to match the request, which we have trained the chatbot with; the chatbot will then respond appropriately.

On the outside, the transcript will look like a normal conversation with two parties responding to each other. The chatbot's function would act as a training tool by receiving input from an individual, processing the input using NLP (Natural Language Processing), and providing an answer (What makes a great chatbot? Laser focus on customers, 2022). The input would be in the form of questions related to matters we usually get from customers and colleagues, pointing them in the right direction or giving them resources that could help them find the answers. The chatbot will be trained to recognise topics and keywords and choose the most appropriate and relevant response using the NLP Nltk module. This tool would make life easier for individuals in positions such as frontline customer support in large and complex organisations.

Motivation

The motivation for this project is to design and construct software that can be used to support employees within an organisation on Human Resources topics. The result will be to fill a gap in most organisations with the rate of resignations and downsizing since the pandemic's beginning. The pressure and toll of employees left within those organisations are struggling. To avoid a high turnover rate focusing on software to assist and train unfamiliar staff or help existing staff is enormously rewarding.

The project motivation cornerstone is based on helping people, and I want to drive the plan's focus towards supporting staff to do their jobs in an efficient and timely manner by creating this tool, building up a knowledge base, and then teaching it to provide the most appropriate answers. The points of everyday user experience outlined by Microsoft (Conversational User Experience Guide, 2021) will allow us to create a solid plan on how we want the chatbot to respond, give it a personality, and map out typical conversations from the user perspective. That will give us an excellent base to set up the appropriate environment and data sets to train the chatbot.

Existing trends with chatbots seem to be driven by customer service by including chatbots in frontline services to improve the organisations' image. You may have had many experiences with speaking to a chatbot already, and an example could be a bank app regarding your account or finding a product on a website that you would like to buy. There does not appear to be much of this technology used in the workplace to support internal staff. AWS and Microsoft have included no-code options within their platforms to allow businesses to deploy their chatbots; however, these options are extremely limited with what they can do, and this project aims to explore open-source libraries and customise as required.

Landscape

Microsoft and AWS provide no code alternatives on their platforms to allow organisations to create a “chatbot” tool that carries out various tasks; however, the reviews indicate a delay in response times and are extremely limited.

Once the knowledge base is input, we would have to add a logic adaptor into the chatbot software and adopt algorithms to allow the chatbot to search google for information or we could create a decision tree which is helpful when there are a lot of variables to consider in an answer.

Database design outline

The database design fundamentals should be determined back at stage 1, as we chose not to proceed with this section, if there is opportunity for a future state, please define the requirements of the database to proceed with an outline of the design. Algorithms

At this stage we will only be working with Keyword matching. If we had more time to design and implement the software, we would consider possibly using several algorithms to allow for effective conversation processing and responses. Possible algorithms considered are String similarity, Combination of algorithms, Spell checker, Natural language parser, Google search.

Business logic implementation

Business logic would include rules determined by the Business regarding data creation, data storage and changes to those files. At this stage we would have to leave this to be determined in the future state.

Web Interface (UML)

The UML is basic but as we have chosen not to proceed with SQL database the UML is not required until those other details are defined.

SQL Database

A database layer will be implemented in the future using SQL. To plan the database, we would also need to consider hibernate mapping, XML mapping, mapping configurations, DAO, Business Logic Layers. As this will be in the future state it would also require us to examine how the UI looks with View information, View feedback, View logs from the database layer. In the future stages we would have to implement the ‘leave feedback’ option within the conversation as a fall back towards the end of the conversation to allow us to review and gauge performance quality. In addition, I do not believe that adding personal information is needed at this stage of the project, if the chatbot were to implement ML/DL algorithms then it would be necessary to personalize the conversation with those details.

Setting up a test that goes through simulated conversations to check expected responses, reviewing and amending the software as part of the testing process will be invaluable.
